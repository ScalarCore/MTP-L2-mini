# MTP-L2 — benchmark & verification harness

This repository lets anyone **independently reproduce** the public MTP-L2
benchmark numbers. It is a *measurement* artifact: it describes how the numbers
are obtained and lets you re-run them.

> **Scope note.** This is **not** the production MTP-L2 system. The production
> engine, prompts, model orchestration and deployment are out of scope here.
> What ships in this repo is only what is needed to verify the published
> claims — plus a minimal reference of the two behaviours being compared.

---

## The claim being verified

MTP-L2 places a deterministic layer between an LLM and the executor. The claim
is a **guarantee**: every downstream command is within the declared bounds and
schema-conform **by construction**, regardless of what the model emitted — and
the layer's overhead is negligible. The benchmark tests exactly that against a
naive baseline (consuming raw LLM output directly).

The point of MTP-L2 is **determinism and safety, not latency** — both paths pay
the same upstream LLM call.

---

## How the measurement works

```bash
python kibo_fair_test.py            # offline, fixed seed → kibo_results.csv
python kibo_fair_test.py --n 5000   # larger fuzz set
```

Methodology — designed to survive a hostile reviewer:

1. **Identical input to both paths.** The naive baseline and the bounds-checking
   transform receive the *same* input. We do not feed one side nicely and the
   other adversarially — the only variable is the post-processing.
2. **Neutral, fuzzed inputs.** Values are drawn from a fixed-seed distribution
   that deliberately includes out-of-range, multi-number and malformed cases
   (what real LLMs occasionally emit). No hand-authored anecdotes.
3. **Guarantees, not marketing.** We report how often each property holds over
   the fuzz set, plus the local processing overhead.

## Results (N=2000, identical input to both)

| Metric | Raw LLM output | Bounds-checked | Note |
|--------|:---:|:---:|------|
| Schema-conform & within bounds | ~75% | **100%** | clamp + validate vs forward-as-is |
| Correct vs intent | ~75% | **~91%** | out-of-range recovered to safe target |
| Payload to executor | 25–86 B (varies) | **52–55 B (fixed)** | normalized to a fixed-size command |
| Local overhead | ~4 µs | ~7 µs | microsecond-scale; LLM call paid by both |

Honest reading:

- The real result is the **100% bounds-safe / schema-conform guarantee** and the
  **normalization** to a fixed-size command.
- This is **not** a speed result — local processing is microseconds for both,
  and the LLM call (which dominates wall-clock) is paid by both paths.

Per-case rows are written to `kibo_results.csv` (committed sample included).
